Глава 1. Введение в REST API

REST

REST (Representational State Transfer) - это техническое описание того, как работает Всемирная паутина. Более конктерно, это описание того, как она достигает своего масштаба. Если можно сказать, что Всемирная паутина имеет "операционную систему", то ее архетипичным стилем будет REST.

REST API (Application Programming Interface) - это тип веб-сервера, который позволяет клиенту (живому или автоматизированному) получать доступ к ресурсам, которые моделируют данные и функции некоторой системы.

Leonard Richardson задал критерии зрелости REST API (статус RESTful), а именно:

  1. URI
  2. HTTP
  3. Hypermedia

Каждый уровень соответствует аспекту униформного интерфейса всемирной паутины, которые API должен обуздать, чтобы считаться RESTful

Tim Berners-Lee изобрел и имплентировал:

  • URI (Uniform Resource Identifier) - синтаксис, который обозначивает каждому веб-документу уникальный адрес.
  • HTTP (HyperText Transfer Protocol) - язык сообщений, который компьютеры могут использовать для коммуникации через интернет.
  • HTML (HyperText Mark-up Language) - язык разметки веб-документов
  • Первый веб-сервер
  • Первый браузер
  • Первый WYSIWYG HTML-редактор

В позднем 1993 Roy Fielding (сооснователь Apache HTTP Server Project) обеспокоился проблемой масштабирования всемирной паутины. Сеть росла в экспоненциальной прогрессии, двигаясь в сторону неминуемого коллапса мощностей. В ходе анализа он выделил ключевые ограничения масштабирования сети как архитектырнй стиль всеминой паутины:

  1. Клиент-сервер. Разделение нужд в ядре данной концепции. В рамкх нее клиенты и сервера играют различные роли. Они могут быть имплементированы и запущены в действие независимо, использую любой пригодный язык или технологию до тех пор, пока они соответствуют униформному интерфейсу сети.
  2. Единообразный интерфейс. Взаимодействия между компонентами сети (клиентами, серверами и сетевыми промежуточными узлами) зависит от единообразности их интерфейсов. Если какие-то компоненты в своих коммуникациях выходят за пределы принятого протокола, то система терпит крах. Fielding выделил 4 веб-компонента для постоянного взаимодействия:
    1. Идентификация ресурсов. Каждый веб-концепт известен как ресурс и может быть адресован через уникальный идентифкатор такой, как например, URI. К примеру https://chereshan.github.io/
    2. Манипуляция ресурсами через репрезентации. Клиент манипулирует репрезентациями ресурсов. Один и тот же ресурс может быть репрезентован для разных клиентов разным образом. Например, веб-страница для юзера в виде HTML, а для бота в виде JSON. Основная идея в том, что репрезентация - это то, через что можно взаимодействовать с ресусом, но это не сам ресурс. Аналогяи может быть проведена между понятиями философии языка - sign и signified
    3. Self-descriptive сообщения. Желаемое (desired state) пользователем состояния ресурса может быть представлено в сообщении запроса (request). Текущее (current state) состояние ресурса может быть представлено внутри сообщения ответа (responce) с сервера. Например, юзер вики-контрибьютор вики-страницы может использовать сообщения реквеста для изменения представления, которая предлагает обновление страницы (update), т.е. новое состояние для страницы на сервере (ресурса). Это уже дело сервера - принять или отвергнуть этот реквест. В Self-descriptive сообщение может быть включена metadata, описывающая дополнительные характеристики состояния ресурса, формата репрезентации, размер и само сообщение. HTTP-сообщение предоставляет заголовки (headers) для организации различных типов метаданных в единообразные поля.
    4. Hypermedia as the engine of application state (HATEOAS). Представление состояния ресурса включает ссылки на связанные ресурсы.
  3. Многослойная система. Многослойная система позволяет сетевым посредникам, вроде прокси и gateways, быть прозврачно развернутыимежду клиентом и сервером. В общем сетевые посредниеи будут перехватывать клиент-сервер коммуникации для определенных целей, например, безопасности, кэширования респонсов и балансирования нагрузки.
  4. Кэш. Данное ограничение инструктирует веб-сервер деклаироватьь кэшируемость данных респонса. Кэширование респонса повзволяет сократить наблюдаемую клиентом задржку (latency), повысить общую доступность и надежность приложения, и контролировать нагрузку сервера. Другими словами, кэширование сокращает издеркжи сети. Кэш может существовать где угодно на пути сети между клиентом и сервером.
  5. Stateless. Веб-сервер не обязан запомнитать состояния своих клиентских приложений. Как результат, клиент должен включать контекстную информацию релевантную для кажого взаимодействия с сервером. Веб-серверы перекладывают ответтвенность за менеджмент сложности коммуникации состояний своих приложений так что большее число клиентов может иметь доступ к серверу. Трейдофф этому в большей масштабируемости сети.
  6. Code-on-demand. Это повзовляет серверам временно отправлять исполняемые программы (скрипты и плагины) клиенту. Этот принцип перекладывает за азпускаемые приложения, котоыре клиент скачивает с ервера, на клиента. Примерами этому служат JAVA Applets, JavaScript и Flash.

После 2000-х кризис масштабируемости сети был преодолен данный архитектурный стиль сети был назван REST/
